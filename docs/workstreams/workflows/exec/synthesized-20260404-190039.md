These are adopted as-is:\n\n1. **5-phase structure**: Parsing → Variables/Conditions → Execution → CLI/Output → Integration Tests\n2. **Core types** in `src/workflow.rs`: `Workflow`, `WorkflowStep`, `OnError` (Stop/Continue/Report), `WorkflowResult`, `StepResult` — matching the SPEC data model exactly\n3. **`load_workflow(path)`** parses YAML, sets `workflow_dir`, resolves recipe paths, validates referenced recipes\n4. **YAML ambiguity check**: pre-parse to `serde_yaml::Value`, reject files with both `steps` and `files`\n5. **`resolve_step_variables()`**: workflow vars → simultaneous `vars_map` rename (snapshot-then-apply) → `vars` overrides\n6. **`evaluate_when()`**: render as Jinja2 with workflow-level vars, falsy = empty / \"false\" (case-insensitive) / \"0\"\n7. **`when` evaluated against workflow-level vars** (before vars_map/vars), per AC-3.1\n8. **Extract `run_recipe()` from `cmd_run`**: refactor existing monolith (lines 274–366) into reusable function\n9. **Single `ExecutionContext`** shared across all steps — `virtual_files` carries across steps for dry-run\n10. **`on_error` semantics**: stop = halt + step's exit code; continue = proceed + exit 0; report = proceed + exit 3\n11. **Per-step `on_error` override**: `step.on_error.unwrap_or(workflow.on_error)`\n12. **Step variable validation at execution time** (not upfront) — skipped steps don't get validated\n13. **CLI**: add `Workflow` variant to `Commands` enum, same global options as `Run`\n14. **Output**: `format_workflow_json()` and `format_workflow_human()` in `output.rs`\n15. **25 fixture directories** matching the SPEC's required test fixtures table\n16. **No new exit codes** — all workflow errors map to existing 1/2/3/4\n\n---\n\n## Disagreements\n\n### D-1: Test Organization\n\n**Plan A**: Inline `#[cfg(test)]` modules in `src/workflow.rs` + extend `tests/cli.rs`\n**Plan B**: Separate `tests/spec/frN_*.rs` files with traceability naming (`spec::frN::ac_*`)\n\n**[HUMAN DECISION NEEDED]**: The SPEC's traceability matrix explicitly uses `spec::frN::*` naming (lines 542–553). Plan B aligns with this convention, but creates more files. Plan A keeps tests co-located with implementation. Options:\n- **(a)** Inline unit tests in `workflow.rs` for internal functions + separate `tests/spec/` modules for integration/AC-traceable tests (hybrid)\n- **(b)** All tests in `tests/spec/frN_*.rs` files (Plan B pure)\n- **(c)** All tests inline in `workflow.rs` + `tests/cli.rs` (Plan A pure)\n\n**Recommendation**: Option (a) — unit tests for `resolve_step_variables` and `evaluate_when` belong inline; AC-traceable integration tests belong in `tests/spec/`.\n\n---\n\n### D-2: `detect_file_type` Location\n\n**Plan A**: Put `detect_file_type()` in `src/workflow.rs`\n**Plan B**: Put it in `src/main.rs`\n\n**Adopted**: `src/workflow.rs` — this is workflow-domain logic (detecting whether YAML is a workflow), and `main.rs` is already large. Both `cmd_validate` and `cmd_vars` call into it.\n\n---\n\n### D-3: Workflow Fixture Test Runner\n\n**Plan A**: Extend existing `tests/integration.rs` fixture runner to auto-detect `workflow.yaml`\n**Plan B**: Create separate `tests/common/workflow_harness.rs`\n\n**[HUMAN DECISION NEEDED]**: Options:\n- **(a)** Extend existing `tests/integration.rs` — one runner, auto-detects `recipe.yaml` vs `workflow.yaml` per fixture directory\n- **(b)** Separate `tests/common/workflow_harness.rs` — dedicated runner for workflow fixtures, imported by spec tests\n\n**Recommendation**: Option (a) — the existing runner already handles `discover_fixtures()`, `setup_existing()`, `diff_expected()`. Adding `workflow.yaml` detection is minimal. Keeps one source of truth for fixture conventions.\n\n---\n\n### D-4: `run_recipe` Fail-Fast Within a Recipe\n\n**Plan A** raises this as `[HUMAN DECISION NEEDED]`: should `run_recipe()` always fail-fast within a single recipe (stop on first operation error), or accept a `fail_fast: bool` parameter?\n\n**Plan B** does not explicitly address this.\n\n**SPEC says** (AC-5.9): \"operations that completed before the failure within that step\" — confirming fail-fast within a step.\n\n**Adopted**: `run_recipe()` always fails fast within a recipe. `on_error` only controls cross-step behavior. This is the only interpretation consistent with AC-5.9.\n\n---\n\n### D-5: `jig validate` Accumulation vs Fail-Fast on Invalid Steps\n\n**Plan B** raises: should `jig validate` accumulate all invalid steps or fail-fast on first invalid step?\n\n**[HUMAN DECISION NEEDED]**: Options:\n- **(a)** Accumulate — validate all steps, report all invalid recipes in one pass (more useful for the caller)\n- **(b)** Fail-fast — stop on first invalid step (simpler, consistent with recipe validate which checks one file)\n\n**Recommendation**: Option (a) — accumulate all invalid steps. Consistent with how variable validation accumulates errors (AC-2.7). An LLM caller benefits from seeing all problems at once.\n\n---\n\n### D-6: `recipe` Field in JSON Output — Original vs Resolved Path\n\n**Plan B** raises: should the `recipe` field in JSON output show the original path from the workflow YAML, the resolved absolute path, or both?\n\n**SPEC says**: The JSON output schema example shows `\"recipe\": \"./model/add-field/recipe.yaml\"` — the original workflow-authored path.\n\n**Adopted**: Use the original path as authored in the workflow YAML. This matches the SPEC schema and is what the LLM caller wrote, so it's recognizable.\n\n---\n\n### D-7: Workflow Run Preflight Depth\n\n**Plan B** raises: should `jig workflow` pre-validate all step recipe structures before step 1 executes, or validate lazily per-step?\n\n**Plan A**: `load_workflow()` validates all recipes at load time (Phase 1.2).\n\n**SPEC says**: AC-1.18 \"WHEN validating a workflow, the system SHALL also parse and validate each referenced recipe file\" — but this is about `jig validate`, not `jig workflow`.\n\n**[HUMAN DECISION NEEDED]**: Options:\n- **(a)** Pre-validate all recipes at load time in `jig workflow` too — fail before executing step 1 if step 3's recipe is broken (prevents partial execution with known-bad recipes)\n- **(b)** Validate lazily per-step at execution time — only load and validate each recipe when its step runs (a skipped step's recipe is never validated)\n- **(c)** Pre-validate unconditional steps only; validate conditional steps lazily\n\n**Recommendation**: Option (a) — pre-validate all recipes. Partial execution of a workflow that will inevitably fail is worse than failing upfront. This also matches Plan A's approach and is consistent with the validate-before-execute principle (AC-6.9 does this for variables).\n\n---\n\n### D-8: `vars_map` Duplicate Target Behavior\n\n**Plan B** raises: what happens with `{a: c, b: c}` — two sources mapping to the same target?\n\n**SPEC says**: AC-4.10 says simultaneous application. Does not specify tie-breaking for duplicate targets.\n\n**[HUMAN DECISION NEEDED]**: When two `vars_map` entries map to the same target (e.g., `{a: c, b: c}`), which source wins? Options:\n- **(a)** Last key in YAML insertion order wins (IndexMap preserves order)\n- **(b)** Error — reject duplicate targets at parse time\n\n**Recommendation**: Option (b) — error. Duplicate targets are likely a mistake, and silently picking one source is surprising. Fail at parse time with a clear message.\n\n---\n\n## Unique Insights\n\n### From Plan A (not in Plan B):\n\n1. **`vars_map` simultaneous rename implementation detail**: Snapshot all source values first into a separate map, then remove all sources, then insert all targets.
