No stage has side effects except operation execution (which writes files).\n     8\t\n     9\t```\n    10\tCLI Input\n    11\t  |\n    12\t  v\n    13\t+------------------+\n    14\t| CLI Parser       |  args, flags, var sources\n    15\t| (clap)           |\n    16\t+------------------+\n    17\t  |\n    18\t  v\n    19\t+------------------+\n    20\t| Recipe Loader    |  YAML -> Recipe struct\n    21\t| (serde_yaml)     |\n    22\t+------------------+\n    23\t  |\n    24\t  v\n    25\t+------------------+\n    26\t| Variable         |  JSON input -> typed, validated Variables\n    27\t| Validator        |  merge: defaults < file < stdin < inline\n    28\t| (serde_json)     |\n    29\t+------------------+\n    30\t  |\n    31\t  v\n    32\t+------------------+\n    33\t| Template         |  template + variables -> rendered string\n    34\t| Renderer         |  custom filters registered here\n    35\t| (minijinja)      |\n    36\t+------------------+\n    37\t  |\n    38\t  v\n    39\t+------------------+\n    40\t| Operation        |  rendered content + file operation spec\n    41\t| Executor         |  -> file writes (or dry-run recording)\n    42\t+------------------+\n    43\t  |\n    44\t  v\n    45\t+------------------+\n    46\t| Output           |  results -> JSON (stdout) or human (stderr)\n    47\t| Formatter        |  TTY auto-detect, --json, --quiet\n    48\t+------------------+\n    49\t  |\n    50\t  v\n    51\tExit Code (0-4)\n    52\t```\n    53\t\n    54\t## Module Map\n    55\t\n    56\t### `src/main.rs` — Entry Point\n    57\t\n    58\tOwns: CLI argument parsing via clap. Wires the pipeline together.\n    59\tDepends on: every other module (it's the composition root).\n    60\t\n    61\tResponsibilities:\n    62\t- Define CLI commands: `run`, `render`, `validate`, `vars`\n    63\t- Parse `--vars`, `--vars-file`, `--vars-stdin`, `--dry-run`, `--json`, `--quiet`, `--force`, `--base-dir`, `--verbose`\n    64\t- Call pipeline stages in order\n    65\t- Map errors to exit codes\n    66\t\n    67\t### `src/recipe.rs` — Recipe Parsing\n    68\t\n    69\tOwns: `Recipe`, `VariableDecl`, `FileOp` structs. YAML deserialization.\n    70\tDepends on: serde, serde_yaml.\n    71\t\n    72\tResponsibilities:\n    73\t- Deserialize `recipe.yaml` into `Recipe` struct\n    74\t- Validate structural correctness operation types: create vs inject vs replace vs patch\n    77\t\n    78\tKey types:\n    79\t```rust\n    80\tstruct Recipe {\n    81\t    name: Option<String>,\n    82\t    description: string, number, boolean, array, object, enum\n    89\t    required: bool,\n    90\t    default: Option<Value>,\n    91\t    description: Option<String>,\n    92\t    values: Option<Vec<String>>,  // 98\t    Inject { template: String, inject: String, mode: InjectMode, skip_if: Option<String> },\n    99\t    Replace { template: String, replace: String, spec: ReplaceSpec, fallback: Fallback },\n   100\t    Patch { template: String, patch: String, anchor: Anchor, skip_if: Option<String> },\n   101\t}\n   102\t```\n   103\t\n   104\t### `src/variables.rs` — Variable Validation\n   105\t\n   106\tOwns: Variable merging, type checking, default application.\n   107\tDepends on: serde_json, recipe types.\n   108\t\n   109\tResponsibilities:\n   110\t- Parse JSON from multiple sources (inline string, file, stdin)\n   111\t- Merge with precedence: recipe defaults < vars-file < vars-stdin < inline --vars\n   112\t- Type-check each value against its `VariableDecl`\n   113\t- Validate required fields are present\n   114\t- Validate enum values against allowed set\n   115\t- Validate array item types\n   116\t- Produce clear error messages with hints (I-4)\n   117\t\n   118\t### filters).\n   122\t\n   123\tResponsibilities:\n   124\t- Create minijinja `Environment` with templates loaded from recipe-relative paths\n   125\t- Register built-in filters: `snakecase`, `camelcase`, `pascalcase`, `kebabcase`, `upper`, `lower`, `capitalize`, `replace`, `pluralize`, `singularize`, `quote`, `indent`, `join`\n   126\t- Render a template with a variables context\n   127\t- Return `Result<String>` — never silently produce wrong output (I-10)\n   128\t\n   129\tDesign note: The renderer is stateless per invocation. Create environment, register filters, load templates, render, return. No caching needed at v0.1 scale.\n   130\t\n   131\t### `src/operations/mod.rs` — Operation Dispatch\n   132\t\n   133\tOwns: `Operation` trait, dispatch logic, dry-run tracking.\n   134\tDepends on: operation implementations, renderer output.\n   135\t\n   136\t```rust\n   137\ttrait Operation {\n   138\t    fn execute(&self, rendered: &str, ctx: &ExecutionContext) -> Result<OpResult>;\n   139\t}\n   140\t\n   141\tstruct ExecutionContext {\n   142\t    base_dir: PathBuf,\n   143\t    dry_run: bool,\n   144\t    force: bool,\n   145\t}\n   146\t\n   147\tenum OpResult {\n   148\t    Success { action: &'static str, path: PathBuf, lines: usize },\n   149\t    Skip { path: PathBuf, reason: String },\n   150\t    Error { path: PathBuf, error: StructuredError, rendered_content: String },\n   151\t}\n   152\t```\n   153\t\n   154\tOperations execute in declaration order (I-9). Each operation receives the already-rendered content — rendering happens before execution, not during.\n   155\t\n   156\t### `src/operations/create.rs` — Create Operation\n   157\t\n   158\tOwns: New file creation logic.\n   159\tDepends on: std::fs.\n   160\t\n   161\tResponsibilities:\n   162\t- Render the `to` path as a template (it can contain `{{ variables }}`)\n   163\t- Create parent directories as needed\n   164\t- Write rendered content to the target path\n   165\t- `skip_if_exists`: if true and target exists, return `OpResult::Skip`\n   166\t- If not `--force` and target exists and `skip_if_exists` is false, return error\n   167\t\n   168\t### `src/operations/inject.rs` — Inject Operation\n   169\t\n   170\tOwns: Content injection into existing files.\n   171\tDepends on: regex.\n   172\t\n   173\tResponsibilities:\n   174\t- Read the target file\n   175\t- `skip_if`: search for string in file content, skip if found (I-2: idempotency)\n   176\t- Find the anchor line using regex pattern\n   177\t- `at: first` (default) or `at: last` — which match to use\n   178\t- Insert rendered content `after` or `before` the anchor line\n   179\t- `prepend`: insert at beginning of file\n   180\t- `append`: insert at end of file\n   181\t- Write the modified content back\n   182\t\n   183\t### `src/operations/replace.rs` — Replace Operation\n   184\t\n   185\tOwns: Region replacement in existing files.\n   186\tDepends on: regex.\n   187\t\n   188\tResponsibilities:\n   189\t- `between`: find start pattern, find end pattern, replace everything between them (exclusive of markers)\n   190\t- `pattern`: find matching line(s), replace them entirely\n   191\t- `fallback`: what to do when pattern not found — `append`, `prepend`, `skip`, `error` (default)\n   192\t- Preserve line endings and file encoding\n   193\t\n   194\t### `src/operations/patch.rs` — Patch Operation\n   195\t\n   196\tOwns: Scope-aware content insertion. Delegates scope detection.\n   197\tDepends on: scope module, regex.\n   198\t\n   199\tResponsibilities:\n   200\t- Find anchor pattern in file\n   201\t- Determine scope boundaries using the scope module\n   202\t- Apply `find` narrowing within the scope\n   203\t- Insert rendered content at the specified position\n   204\t- `skip_if`: idempotency check within the scope\n   205\t\n   206\t### `src/scope/mod.rs` — Scope Detection Dispatch\n   207\t\n   208\tOwns: Scope type classification, dispatch to indent vs delimiter strategies.\n   209\tDepends on: indent, delimiter, position submodules.\n   210\t\n   211\t```rust\n   212\tenum ScopeType {\n   213\t    Line,              // just the matched line\n   214\t    Block,             // indent-based block\n   215\t    ClassBody,         // indent-based class body\n   216\t    FunctionBody,      // indent-based function body\n   217\t    FunctionSignature, // delimiter: ( to )\n   218\t    Braces,            // delimiter: { to }\n   219\t    Brackets,          // delimiter: [ to ]\n   220\t    Parens,            // delimiter: ( to )\n   221\t}\n   222\t```\n   223\t\n   224\t### `src/scope/indent.rs` — Indentation-Based Scope Detection\n   225\t\n   226\tOwns: Scope detection for indentation-significant languages (Python, YAML).\n   227\t\n   228\tAlgorithm:\n   229\t1. Find the anchor line's indentation level\n   230\t2. Walk forward: scope includes all lines indented deeper than the anchor\n   231\t3. Scope ends when indentation returns to the same or shallower level\n   232\t4. Handle blank lines within the scope (they don't end it)\n   233\t\n   234\t### `src/scope/delimiter.rs` — Delimiter-Based Scope Detection\n   235\t\n   236\tOwns: Scope detection for delimiter-based languages (C-family, JSON, TypeScript).\n   237\t\n   238\tAlgorithm:\n   239\t1. From the anchor line, find the opening delimiter\n   240\t2. Count nesting depth (handle nested delimiters)\n   241\t3. Scope is everything between opening and closing delimiter (exclusive)\n   242\t4. Handle string literals (don't count delimiters inside strings)\n   243\t\n   244\t### `src/scope/position.rs` — Semantic Position Heuristics\n   245\t\n   246\tOwns: Position determination within a scope.\n   247\t\n   248\t| Position | Heuristic |\n   249\t|----------|-----------|\n   250\t| `before` | First line of scope |\n   251\t| `after` | Last line of scope |\n   252\t| `before_close` | Line before closing delimiter or dedent |\n   253\t| `after_last_field` | Last line matching `^\\s+\\w+\\s*[:=]` |\n   254\t| `after_last_method` | Last line matching `^\\s+def \\w+` |\n   255\t| `after_last_import` | Last line matching `^\\s*(from\\|import) ` |\n   256\t| `sorted` | Alphabetical insertion among siblings |\n   257\t\n   258\t### `src/filters.rs` — Custom Jinja2 Filters\n   259\t\n   260\tOwns: Filter function implementations and registration.\n   261\tDepends on: heck (case conversions), minijinja (filter API).\n   262\t\n   263\tFilters are pure functions: `fn(value: &str, args...) -> String`. No side effects, no I/O. This module is the thinnest — most filters are one-liners wrapping `heck` or standard string operations.\n   264\t\n   265\t### `src/output.rs` — Output Formatting\n   266\t\n   267\tOwns: Dual-stream output (I-8), TTY detection.\n   268\tDepends on: serde_json, owo-colors.\n   269\t\n   270\tResponsibilities:\n   271\t- Collect `Vec<OpResult>` from operation execution\n   272\t- JSON mode: serialize results to stdout\n   273\t- Human mode: colored, summarized output to stderr\n   274\t- Auto-detect: if stdout is a TTY, use human mode; if piped, use JSON\n   275\t- `--json`: force JSON\n   276\t- `--quiet`: suppress all output except errors\n   277\t- `--verbose`: include rendered content in output\n   278\t\n   279\t### `src/error.rs` — Structured Error Types\n   280\t\n   281\tOwns: Error types, exit code mapping, error formatting.\n   282\tDepends on: nothing (leaf module).\n   283\t\n   284\t```rust\n   285\tenum JigError {\n   286\t    RecipeValidation(String),     // exit 1\n   287\t    TemplateRendering(String),    // exit 2\n   288\t    FileOperation(String),        // exit 3\n   289\t    VariableValidation(String),   // exit 4\n   290\t}\n   291\t```\n   292\t\n   293\tEvery error includes: what, where, why, hint (I-4). File operation errors include the rendered content so the caller can fall back to manual editing (I-10).\n   294\t\n   295\t### `src/workflow.rs` — Multi-Recipe Orchestration (v0.3+)\n   296\t\n   297\tOwns: Workflow definition, step execution, conditional logic, variable mapping.\n   298\tDepends on: recipe, variables, renderer, operations.\n   299\t\n   300\tNot built until v0.3. The operation receives a plain string, not a template. This means:\n   345\t- Rendering errors are caught before any file is touched\n   346\t- The rendered content is always available for error messages (I-4)\n   347\t- Operations are simpler — they just deal with strings and files\n   348\t\n   349\t### D-2: Recipe-Relative Template Resolution\n   350\t\n   351\tTemplate paths in recipes are relative to the recipe file, not the working directory. This keeps recipes self-contained (I-7). The `--base-dir` flag only affects output paths (where files are written), not template paths (where templates are read from).\n   352\t\n   353\t### D-3: Regex for Anchoring, Not Parsing\n   354\t\n   355\tAnchors use regex, not AST parsing. This is deliberate:\n   356\t- Language-agnostic (same tool works for Python, TypeScript, Go, Rust)\n   357\t- No parser dependencies (keeps binary small, I-6)\n   358\t- Failure is explicit (regex didn't match = clear error, not a silent wrong parse)\n   359\t- Scope detection adds just enough structure without full parsing\n   360\t\n   361\t### D-4: Operations Are Values, Not Traits (Initially)\n   362\t\n   363\tFor v0.1, operations can be a simple enum with match dispatch rather than a trait-object pattern. The trait abstraction is only needed if we add user-defined operations, which isn't planned. Keep it simple — an enum with four variants covers all cases through v0.4.\n   364\t\n   365\t### D-5: No Global State\n   366\t\n   367\tNo singletons, no global config objects, no lazy_static. Every function takes its inputs as arguments. This makes testing trivial and ensures determinism (I-1).\n   368\t\n   369\t## Iterative Build Strategy\n   370\t\n   371\tEach phase produces a working, testable, demoable artifact.
